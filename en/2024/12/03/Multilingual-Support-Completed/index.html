Multilingual Support Completed

This blog has completed multilingual construction. Readers can click the “🌐” button in the navigation bar to switch languages. I will try my best to ensure that blog posts in various languages ​​are updated synchronously. If you find that an article does not have a corresponding language version, you can use the corresponding template to publish an issue in our GitHub community, or you can also help translate.

The multilingual support (i18n) of the Hexo framework itself only provides internationalization of menus, etc., and cannot achieve multilingual support for blog content. Some themes have this function built in, but the Fluid theme used in this site is unfortunately not on this list. Therefore, I can only find relevant solutions on the Internet. I will record my findings and solution steps in this article, hoping to help other readers who are willing to build an internationalized blog.

Maintain multiple sets of Hexo projects, integrate other languages ​​in the main language public folder (the chosen of this site)

This method was discovered in the blog “Brando的自留地”. Since the original text (web archive) just introduces the solution available when using Fluid theme, and the principle is easy to understand, this site chose this method to achieve the internationalization of the blog.

Preliminary preparation

According to the idea of ​​the original text, copy the files of the main site (except node_modules) multiple times, and each copy corresponds to a language version. For example, this site supports three languages, ​​​​Simplified Chinese, Traditional Chinese (Taiwan) and English, that is, save the original Simplified Chinese project in two copies. Execute npm install in the newly added language branch to install dependencies.

Then, copy the articles under main-zh-CN/source to other language branches and translate them according to the target language. This completes the basic steps of “extralocalization”.

Integrate other language articles

Next, we are ready to deploy the site. Generate public files under each language branch, switch to the main language branch, and create two new folders, en and zh-TW, under main-zh-CN/public. Note that since the addresses of each language branch are hard-coded in the theme configuration, these two folders must be named according to the set path. Copy the public files under the other two language branches respectively. Then, deploy directly.

We can find that the public file generated by Hexo is the entire website structure itself. Switching to other languages ​​means accessing the subdirectory of the website root directory (locally, public), that is, /en/ or /zh-TW/. If we jump out of the internationalization construction itself, we can also use this to perform more free and personalized operations. For example, when you write an html document by hand and want to connect it to a website built with the Hexo framework, you can start directly from public. For example, if I want to use a homepage instead of the one generated by Fluid, I can replace public/index.html; if I want to add a custom login/subscription page, I can create a new accounts/subscribe subdirectory in it, and access it in the browser. The URL is https://example.com/accounts/page.html.

But please note that the source files of these custom projects must be saved elsewhere, otherwise, when the hexo clean command is used to clean the cache, all the hard work under public will be burned.

Regarding this “free” custom operation, as well as the topic of related automated maintenance and deployment, I will continue to study related solutions. Currently, I am considering writing a Python program to implement these operations.

Disadvantages

The most obvious one is: maintaining multiple sets of projects at the same time, the operation is complicated, and it is easy to “lose one thing while taking care of another”. Especially when changing the blog configuration file, it is more likely to have problems.

Another one is the common problem of multi-language sites: keeping the synchronous update of multi-language content is indeed a very test of the blogger’s ability and perseverance.

Maintain a single Hexo project and deploy multiple repositories

The related page of The blog “井说” (Web Archive) introduces another method and provides an automated deployment solution. This multi-language solution only requires maintaining one set of Hexo projects, but it still requires managing configuration files in different languages.

The author takes the approach of directly creating directories for storing articles in different languages ​​in source to distinguish articles in different languages. In automated deployment, files belonging to different language branches are pushed to different repositories respectively, and the address feature of GitHub Pages is used to distinguish different language versions.

The essential difference between this solution and the previous one is that the previous solution puts “multiple GitHub repositories” locally, that is, multiple Hexo projects; while this solution hosts each project on GitHub in an automated way, and only one set of projects is kept locally to accommodate files of each language branch.

Since this blog does not adopt this solution, it will not be elaborated in detail. However, the original article ends with an idea of ​​using Google Translate’s API to translate articles during deployment. As a “lazy” way to handle multi-language content, this idea is worth exploring (laughs).

But this solution still has very serious flaws: special pages, such as the about page, cannot be internationalized; in addition, articles in other languages ​​can still be seen on the archive page. Perhaps the hexo-generator-i18n plugin can solve this problem, but I have not yet figured out how it works from the documentation and practice; coupled with concerns about potential problems that may be caused by the modification of the main source code, this solution was quickly abandoned by this site after a short period of use.

Written at the end

When studying the third solution, I could hardly hold back my complaints: Why is the ejs code so dazzling?

The internationalization solution of the Hexo framework is still immature, and each theme has its own way, and the Fluid theme unfortunately does not have built-in multilingual content support. I have considered tearing down the entire architecture and starting over, but considering the already huge sunk costs, I chose to stick to it.

The current solution is still immature. For example, when switching languages, readers will only be directed to the homepage of the corresponding language, the local maintenance process management lacks systematization, etc. When I come up with a complete solution, I will publish it in this blog.

In the process of tossing, I came into contact with a lot of new knowledge, discovered other blogs that insist on updating high-quality content, and found more directions worth exploring (tossing). For example, the previously mentioned automated deployment, access to the Google Translate API, and embedded JavaScript. The new gameplay of the public folder seems to be a very rich mine, but according to the information I have found, this mine does not seem to have attracted people’s attention (after all, why write HTML yourself when you can solve it with a framework). I may publish follow-up research on these topics in the future, hoping to provide some insignificant help to others.
